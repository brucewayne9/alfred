  <action>
Create `scripts/backup/weekly_backup.py` as a standalone executable script (chmod +x, shebang `#!/usr/bin/env python3`).

Add `sys.path.insert(0, ...)` for project root imports.

**Main flow (`backup_server(server: dict) -> dict`):**

For each server in SERVERS:
1. Create staging dir: `/tmp/alfred-backups/weekly/YYYY-MM-DD/{server_name}/`
2. **First: collect everything daily does** — reuse the same config collection logic (crontabs, env files, systemd units, database dumps). Import and call the relevant collection functions or replicate the per-server collection logic from DAILY_TARGETS.
3. **Then: collect weekly extras from WEEKLY_EXTRAS:**
   - Package list: `dpkg --list > {staging}/packages.txt` (all servers)
   - Docker volume list: `docker volume ls > {staging}/docker-volumes.txt` (servers with Docker)
   - Docker image list: `docker image ls > {staging}/docker-images.txt` (servers with Docker)
   - Docker volume exports: For key named volumes, export via `docker run --rm -v {volume}:/data -v /tmp:/backup alpine tar czf /backup/{volume}.tar.gz /data`. Target specific volumes, NOT all volumes — prioritize database volumes and app data volumes. Discovery approach: run `docker volume ls -q` at backup time, filter to named volumes (skip anonymous hash-named volumes), export each. Set a per-volume timeout of 300 seconds (5 min) to avoid hanging on huge volumes.
   - For alfred-labs (105): Copy key SQLite DBs from `data/` (conversations.db, learning.db), ChromaDB directory
   - For labsliveserver (104): Be selective — 55 containers means many volumes. Only export volumes matching patterns: `*_db_*`, `*_data_*`, `*postgres*`, `*mysql*`, `*redis*`, `*mongo*`. Skip volumes with "cache", "tmp", "log" in the name.
4. Pack staging dir into `{server_name}-weekly-YYYY-MM-DD.tar.gz`
5. Upload tarball to Drive via `upload_backup(tarball_path, server_name, "weekly")`
6. Clean up staging dir
7. Return result dict (same format as daily)

**After all servers backed up — run retention cleanup:**

Call `prune_old_backups(retention_days=30)` from drive_manager. This prunes BOTH daily and weekly folders older than 30 days across all servers. Log the pruning summary.

**Main function (`main()`):**

1. Configure logging to both stdout and `data/backup_weekly.log` (append, timestamps)
2. Log start time
3. Iterate SERVERS, call `backup_server()` for each, collect results
4. Run retention cleanup: `prune_old_backups(30)`
5. Log summary: servers succeeded/failed, total size, folders pruned
6. Exit 1 if any server failed
7. Log total duration

**Docker volume export strategy (addressing integration risk #1):**
- Do NOT rely on audit.py's volume list (it doesn't collect volumes)
- Discover volumes LIVE at backup time: `docker volume ls -q`
- Filter: skip anonymous volumes (64-char hex names), skip volumes with "cache"/"tmp"/"log" in name
- Export each named volume via Alpine tar container
- Per-volume timeout of 300s prevents hanging
- If a volume export fails, log warning and continue (non-fatal)

**Backup target selection (addressing integration risk #2):**
- Define backup targets by explicit directory paths and volume name patterns, NOT by relying on SQLite file discovery from audit.py
- The WEEKLY_EXTRAS in backup_utils.py defines what to collect per server

**Error handling:** Same as daily — per-server isolation, database dump failures non-fatal, Drive upload failures fatal per-server. Additionally, per-volume-export failures are non-fatal.
