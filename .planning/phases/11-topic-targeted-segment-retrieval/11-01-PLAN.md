<objective>
Build the retrieval core for topic-targeted segment search: merge raw transcript segments into 20-40s windows, embed each window with bge-m3 into a shared ChromaDB collection, and provide a ranked source-scoped query function. Wire embedding to run inline at the end of transcription so every newly-transcribed source is immediately searchable, with lazy backfill for sources transcribed before this phase.

Purpose: TOPIC-01 — the operator must get a ranked list of matching source segments with in/out timestamps and relevance scores. Precision is the make-or-break risk of the whole milestone, and windowing-before-embedding is the design choice that makes it work.
Output: core/forge/search.py (new), an inline embed hook in ingest.py, and tests.
</objective>

  <files>core/forge/search.py</files>
  <action>
Create the new module `core/forge/search.py`. It owns all vector-retrieval logic. Reuse the live stack — NO new dependencies.

Collection accessor:
- `_COLLECTION_NAME = "forge_segments"` — a SINGLE shared collection, source scoped via the `source_id` metadata field (NOT one collection per source — research §3).
- `_get_collection()`: build an `OllamaEmbeddingFunction(url="http://localhost:11434", model_name="bge-m3:latest", timeout=120)` (import from `chromadb.utils.embedding_functions`) and call `get_client().get_or_create_collection(name=_COLLECTION_NAME, metadata={"hnsw:space": "cosine"}, embedding_function=ef)`. Import `get_client` from `core.memory.store`. Pin bge-m3 — never a :cloud model (CLAUDE.md forbids deleting bge-*; research pitfall 5).

Windowing (the precision crux — research §4):
- `build_windows(segments, target_dur_s=30.0, max_dur_s=45.0, min_dur_s=10.0) -> list[dict]`: grow a buffer of adjacent segments; flush the buffer into a window when adding the next segment would exceed `max_dur_s` AND the buffer is already >= `min_dur_s`; emit the trailing buffer at the end. Each window dict carries: source_id, seq_start (first seg's seq), seq_end (last seg's seq), start_s (first), end_s (last), text (segment texts joined with single spaces, stripped), speaker (dominant via collections.Counter over `s.get("speaker") or ""`), win_id (set by caller).
- Do NOT split on speaker boundaries by default — a short interjection flips speaker but not topic; the duration guard absorbs it (research §4).
- win_id format: `f"{source_id}_w{seq_start:04d}"` — stable + idempotent across re-embeds.

Upsert:
- `upsert_windows(source_id, windows)`: `col.upsert(ids=[w["win_id"]...], documents=[w["text"]...], metadatas=[{source_id, start_s, end_s, speaker: (w.get("speaker") or ""), seq_start, seq_end}...])`. CRITICAL: coerce speaker None->"" — ChromaDB metadata rejects None (research pitfall 2).

embed_source_windows (the entry point ingest.py + the endpoint call):
- `embed_source_windows(source_id)`: load segments via `ingest.get_segments(source_id)` (import core.forge.ingest lazily to avoid a circular import); if empty, return 0. Attach source_id to each seg dict if missing. Build windows, set each `win_id`. BEFORE upserting, `col.delete(where={"source_id": {"$eq": source_id}})` to clear orphan win_ids from any prior windowing (research pitfall 4). Then `upsert_windows`. Return the window count.

has_windows:
- `has_windows(source_id) -> bool`: `col.get(where={"source_id": {"$eq": source_id}}, limit=1)` and return `len(res["ids"]) > 0`.

Query (research §3):
- `search_segments(source_id, query, top_k=10, speaker=None, score_threshold=0.45) -> list[dict]`: get collection; if `col.count() == 0` return []. Build `where = {"source_id": {"$eq": source_id}}`; if speaker truthy, `where = {"$and": [{"source_id": {"$eq": source_id}}, {"speaker": {"$eq": speaker}}]}`. Call `col.query(query_texts=[query], n_results=min(top_k, col.count()), where=where, include=["documents","metadatas","distances"])`. For each hit: `score = round(1.0 - dist/2.0, 4)` (cosine dist in [0,2] -> score in [0,1]); skip if `score < score_threshold`; build result {start_s, end_s, text (the document), speaker, score, seq_start, seq_end}. Return sorted by score descending. NEVER return raw distance (research pitfall 7).
