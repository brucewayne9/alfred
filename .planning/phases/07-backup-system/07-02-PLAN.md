  <name>Task 1: Create daily config backup script with SSH collection and Drive upload</name>
  <files>scripts/backup/daily_backup.py</files>
  <action>
Create `scripts/backup/daily_backup.py` as a standalone executable script (chmod +x, shebang `#!/usr/bin/env python3`).

Add `sys.path.insert(0, os.path.join(os.path.dirname(os.path.abspath(__file__)), '..', '..'))` for project root imports.

**Main flow (`backup_server(server: dict) -> dict`):**

For each server in SERVERS:
1. Create staging dir: `/tmp/alfred-backups/daily/YYYY-MM-DD/{server_name}/`
2. Collect config files using `collect_remote_files()` from backup_utils for static file paths in DAILY_TARGETS
3. For command-based collections (crontab dumps, pg_dumpall, mysqldump, redis-cli BGSAVE), run them via `run_cmd()` and write output to staging dir:
   - `crontab -l > {staging}/crontab.txt`
   - `systemctl list-unit-files --type=service --state=enabled > {staging}/systemd-enabled.txt`
   - For alfred-labs (105): `pg_dumpall -U postgres > {staging}/postgresql-dump.sql` (use sudo if needed)
   - For labs-edge (100) and labsliveserver (104): `mysqldump --all-databases > {staging}/mysql-dump.sql` (adapt credentials from server env)
4. Pack staging dir into `{server_name}-daily-YYYY-MM-DD.tar.gz`
5. Upload tarball to Drive via `upload_backup(tarball_path, server_name, "daily")`
6. Clean up staging dir for this server
7. Return result dict: `{"server": server_name, "success": bool, "files_collected": int, "tarball_size": str, "drive_id": str, "error": str | None}`

**Main function (`main()`):**

1. Configure logging to both stdout and a log file at `data/backup_daily.log` (append mode, with timestamps)
2. Log start time
3. Iterate SERVERS, call `backup_server()` for each, collect results
4. Log summary: servers succeeded, servers failed, total size uploaded
5. If any failures, exit with code 1 (so cron notices)
6. Log total duration

**Error handling:**
- Each server is independent — one server failure does not abort others
- Wrap each server's backup in try/except, log the error, continue to next
- Database dump failures are non-fatal (log warning, skip that artifact, still collect other files)
- Drive upload failures ARE fatal for that server (no point keeping local tarball)

**Database dump specifics:**
- PostgreSQL (105 only): `sudo -u postgres pg_dumpall 2>/dev/null` — runs locally, redirect stderr to avoid noise
- MySQL (100, 104): `mysqldump --all-databases --single-transaction 2>/dev/null` — note: may need credentials from env. If dump fails with auth error, log and skip (don't block the whole backup)
- Redis (105 only): Copy `/var/lib/redis/dump.rdb` directly (it's the RDB file). Use `sudo cp` if permission denied.

**Important:** Do NOT attempt to dump Docker volumes in the daily script — that's weekly only. Daily collects configs, env files, crontabs, systemd units, and database dumps only.
